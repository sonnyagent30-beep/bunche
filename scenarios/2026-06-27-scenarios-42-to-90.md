## SCENARIO 43: Double Payment (Accidental Duplicate)

**Customer:** Ada | **Status:** Clicks pay twice → 2 separate Flutterwave transactions

---

### What happens

Customer clicks Flutterwave payment link twice rapidly → two separate bank transfers:
- Transaction A: ₦6,500 (first click)
- Transaction B: ₦6,500 (second click)

Flutterwave sends two separate webhook POSTs to `/webhook/payment`:
- Webhook 1: tx_ref=ORD-20260627-0947, transaction_id=TXN-A, amount=₦6,500
- Webhook 2: tx_ref=ORD-20260627-0947, transaction_id=TXN-B, amount=₦6,500

---

### Bunche system (invisible)

```
Webhook 1 received → Redis SETNX idempotency:TXN-A = 1 → ✅ stored
  → PostgreSQL check: no existing record for TXN-A → ✅
  → Process payment: order ORD-20260627-0947 status = paid ✅

Webhook 2 received → Redis SETNX idempotency:TXN-B = 1 → ✅ stored
  → PostgreSQL check: no existing record for TXN-B → ✅ PROCESSING...
  → DUPLICATE DETECTED: tx_ref matches existing paid order
  → Flutterwave refund API: POST /refunds { transaction_id: TXN-B }
  → Update: refunds table, status=pending
  → Log: double_payment_detected, auto_refund_initiated, TXN-B
```

**Bunche reply (to customer — automatic):**

```
✅ Payment received — ₦6,500 for ISP UK.

⚡ Generating your IP now...
```

---

### Second payment auto-refund

Bunche's Flutterwave integration auto-initiates refund for TXN-B → Flutterwave processes → customer receives ₦6,500 back within 5-7 business days.

**Bunche reply (separate message):**

```
Hi Ada — we noticed you sent two payments for the same order.
We've automatically refunded the second payment of ₦6,500.

Refund reference: FLW-REF-DOUBLE-[TIMESTAMP]
⏰ It will land in your account within 5-7 business days.

Your first payment is being processed — your proxy will arrive shortly ⚡
```

**Admin side:** 🟢 Logged: `double_payment`, auto_refund_txn_b, customer_notified

---

## CRITICAL SECURITY NOTE

Flutterwave tx_ref is customer-controlled input. The tx_ref format `ORD-YYYYMMDD-XXXX` is predictable. In Scenario 46 (amount tampering), we address this. For double-payment, the idempotency key is the Flutterwave `transaction_id` (Flutterwave-controlled), NOT `tx_ref` — so this scenario is protected.
